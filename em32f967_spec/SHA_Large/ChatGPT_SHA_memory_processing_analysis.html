🧩 一、整體呼叫鏈

呼叫順序如下：

scss
Application
   ↓
rts5912_sha256_handler()
   ↓
rts5912_sha256_update()
   ↓
rts5912_sha256_process()

🧠 二、逐層行為解析

### (1) handler 層

c
memcpy(rts5912_sha256_ctx->sha2_data_in_sram, pkt->in_buf, pkt->in_len);
ret_val = rts5912_sha256_update(ctx->device,
                                rts5912_sha256_ctx->sha2_data_in_sram,
                                pkt->in_len);

這裡的動作是：

  • 直接把 pkt->in_buf（application 的輸入資料）整包複製到 sha2_data_in_sram。

  • 然後呼叫 update()。

⚠️ 問題：
如果 pkt->in_len = 391 KB，而 sha2_data_in_sram 是放在 SRAM（假設大小不到 272 KB），
→ 這個 memcpy 就會爆掉。

這是最關鍵的設計錯誤點。

### (2) update 層

c
while (len >= RTS5912_SHA2DMA_BLOCK_SIZE) {
    blk_size = len >> RTS5912_SHA2DMA_BLOCK_SHIFT;
    ret_val = rts5912_sha256_process(dev, input, blk_size);
    ...
}

這層在做的是：

  • 將輸入資料切成「SHA block（一般是 64 bytes）」為單位。

  • 每次呼叫 rts5912_sha256_process() 去交給硬體運算。

🟢 優點：
這層確實有分段處理邏輯（由硬體分段運算，不會一次全吃完）。

🔴 缺點：
它仍然假設「input 指向的是完整的、可隨時訪問的 buffer」。
換句話說，它不會去分批 memcpy application data；
所以若 handler 層一次丟進 391 KB，它也會假設這個 391 KB 已經全在 SRAM 內。

### (3) process 層

c
sha2dma_regs->sar = (uint32_t)(&(input[idx]));
...
sha2dma_regs->ctrl_high = blk_size << RTS5912_SHA2DMA_8Byte_SHIFT;
sha2dma_regs->channel_enable = ...

這裡可明確看到：

  • 它把 input[idx] 的實體位址（SAR）交給 SHA2 DMA 控制器。

  • 也就是說硬體會直接 DMA 從 input buffer 讀資料。

⚠️ 這代表一個前提：

input 的實體記憶體必須是 DMA 可訪問的連續空間（physical contiguous），
而且在 DMA 期間不能被 CPU 改動。

🔍 三、綜合記憶體與流程分析

層級功能是否有分段是否複製整包是否會爆RAM
handler上層入口❌ 無✅ 整包 memcpy⚠️ 會
update分塊處理✅ 有❌ 不再 memcpy—（但前面已爆）
processDMA 傳輸✅ 有❌ 不複製DMA需求正常

結論：

  • update() 與 process() 寫得不錯，有分段設計。

  • 但 handler() 直接把整個資料 memcpy 到 SRAM，導致架構上無法支援大檔案（> SRAM buffer）。

🧮 四、推測硬體意圖

從 rts5912_sha256_process() 來看：

c
for (; blk_size > 0; blk_size -= RTS5912_SHA2DMA_MAXIMUM_BLOCK_NUM)

→ 表示硬體支援「一次 DMA 處理多個 block」，例如：

  • RTS5912_SHA2DMA_MAXIMUM_BLOCK_NUM 可能是 255；

  • 每個 block 64 bytes；

  • 所以一次 DMA 可以吃到 255 × 64 = 16 320 bytes ≈ 16 KB。

也就是：
✅ 硬體一次可處理約 16 KB。
❌ 但你的 handler 卻把 391 KB 全丟進 SRAM。

✅ 五、正確做法（改良版）

最合理的方式，是讓 handler() 改為分段 memcpy + 多次 update()：

c
#define CHUNK_SIZE 16384  // 跟硬體DMA對齊，例如16KB

static int rts5912_sha256_handler(struct hash_ctx *ctx, struct hash_pkt *pkt, bool finish)
{
    struct rts5912_sha256_context *rts5912_sha256_ctx =
        (struct rts5912_sha256_context *)(ctx->device)->data;
    int ret_val = 0;
    size_t processed = 0;

    while (processed < pkt->in_len) {
        size_t chunk = MIN(CHUNK_SIZE, pkt->in_len - processed);

        memcpy(rts5912_sha256_ctx->sha2_data_in_sram,
               pkt->in_buf + processed,
               chunk);

        ret_val = rts5912_sha256_update(ctx->device,
                                        rts5912_sha256_ctx->sha2_data_in_sram,
                                        chunk);
        if (ret_val) return ret_val;
        processed += chunk;
    }

    if (finish)
        ret_val = rts5912_sha256_finish(ctx->device, pkt->out_buf);

    return ret_val;
}

這樣：

  • 每次只複製 ≤16 KB；

  • update() 再分割給 process()；

  • 完全不會爆 SRAM；

  • 對 application 來說仍是「一次呼叫 hash_compute()」。

💡 六、額外觀察（進階）

你可以看出 process() 在最後：

c
for (uint32_t i = 0; i < 8; i++) {
    rts5912_sha256_ctx->state[i] = sha2_regs->digest[i << 1];
}

這代表它每次 DMA 後會讀回硬體 digest 暫存值，
並儲存在 ctx->state[] 中，表示它支援「連續 update」邏輯。
👉 所以分多次呼叫 update() 是正確且安全的，
這符合 SHA-256 streaming 的標準演算法模型。
